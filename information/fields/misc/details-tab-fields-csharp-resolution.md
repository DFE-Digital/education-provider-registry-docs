# Details Tab Fields C# Resolution

## Purpose

We need a stronger way to understand which fields can be shown on the public GIAS Details tab for each establishment type.

`docs/data/extract-data/establishment-fields/establishment-details-tab-fields-by-type.md` was built from one rendered public page per establishment type. That is useful public UI evidence, but it can miss fields where the sampled establishment is not representative.

This note resolves the Details-tab field surface from the C# frontend and legacy field/type metadata. It should be read alongside the live-page scrape, not as a replacement for it.

## Methodology

Sources used:

- `get-information-about-schools/Web/Edubase.Web.UI/Areas/Establishments/Views/Establishment/Partials/TabDetails.cshtml`
- `get-information-about-schools/Web/Edubase.Web.UI/Areas/Establishments/Views/Establishment/Partials/Header.cshtml`
- `get-information-about-schools/Web/Edubase.Services/Establishments/DisplayPolicies/EstablishmentDisplayEditPolicy.cs`
- `get-information-about-schools/Web/Edubase.Services/Establishments/Models/EstablishmentFieldListBase.cs`
- `get-information-about-schools/Web/Edubase.Services.Texuna/Establishments/EstablishmentReadApiService.cs`
- `docs/data/extract-data/establishment-fields/EstablishmentFieldToTypeMapping.csv`
- `docs/data/extract-data/establishment-fields/establishment-details-tab-fields-by-type.md`

The resolution was produced as follows:

1. Read the C# Details-tab Razor partials to identify the labels that the public Details tab has code to render.
2. Read `EstablishmentDisplayEditPolicy` and `EstablishmentFieldListBase` to connect labels to display-policy properties.
3. Confirmed that the C# frontend retrieves display policy from the Java/backend API using `establishment/{urn}/display-policy`.
4. Used `EstablishmentFieldToTypeMapping.csv` as the type-applicability input for display-policy-backed fields.
5. Added a small number of header fields that are rendered unconditionally by the C# view, such as `Address`, `Local authority`, `ID`, `Establishment status`, `Open date` and `Reason establishment opened`.

## Key Finding

The C# code does encode the renderable Details-tab labels and the conditions under which each row is emitted. It does not, by itself, contain a complete hard-coded per-establishment-type field matrix.

The per-record booleans come from the backend display-policy endpoint:

`establishment/{urn}/display-policy`

That means the public Details-tab result is the combination of:

- C# Razor rows and labels
- C# type-sensitive labels, such as `Manager` for children's centres and `Headteacher/Manager` for online providers
- Backend display-policy booleans
- Legacy field/type metadata
- Record-specific data and relationships
- User/public access rules
- Feature flags and special-case suppression rules

This explains why extract population and field/type mapping can still produce false positives. For example, URN `100759` has a proprietor value in the extract, but the public Details tab does not render `Proprietor's name`.

The table below is a conservative resolver output. It answers:

> Which Details-tab labels have C# render points and are supported by establishment-type field metadata?

It does not answer:

> Which labels definitely render on every public page for that establishment type?

For that stronger claim, we still need either the actual display-policy response for representative URNs or public-page checks.

## Important Limitations

This is a C# and metadata resolver, not a live public-page scrape.

Some fields are not resolved here because the C# row is not driven only by `EstablishmentFieldToTypeMapping.csv`:

- `Website` depends on `Contact_WebsiteAddress`.
- `Ofsted rating and reports` depends on Ofsted link/rating logic, not just a simple field/type mapping.
- `Data from other services` depends on service-link logic.
- Academy trust and academy sponsor labels are produced through group relationships in `Header.cshtml`, not a simple establishment field row.
- `Proprietor's name` is a normal C# Details-tab row, but the live URN `100759` check shows it can be suppressed by public display policy even where extract data is populated.
- Secure and legacy types can have broader metadata mappings than their public Details-tab page actually shows.

## Recommendation

Use this document as the field-surface resolver.

Use `establishment-details-tab-fields-by-type.md` as rendered public-page evidence.

Use `establishment-details-tab-fields-extract-gap-analysis.md` to target specific URNs for public checks where the resolver and the observed sample disagree.

We should only update the observed Details-tab catalogue when either:

- A public page for a captured URN confirms the field label renders, or
- We capture the actual `display-policy` response for that URN and confirm the relevant display-policy boolean is true.
